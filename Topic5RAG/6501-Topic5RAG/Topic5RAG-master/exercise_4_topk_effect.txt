LATENCY COMPARISON
======================================================================

Average Response Times:
- k=1:  3.86s  (baseline)
- k=3:  4.10s  (+6% vs k=1)
- k=5:  4.45s  (+15% vs k=1)
- k=10: 5.23s  (+35% vs k=1)
- k=20: 6.74s  (+75% vs k=1)

Observation: Response time scales non-linearly with k due to:
1. More chunks to retrieve from FAISS
2. Longer context window for LLM to process
3. More tokens to generate with larger context

======================================================================
ANALYSIS
======================================================================

Key Findings on Top-K Effect:

1. k=1:
   - Fastest response, but may miss important context
   - Risk of incomplete answers if top result lacks full information
   - Good only for very specific, simple factual lookups

2. k=3-5: (RECOMMENDED RANGE)
   - Good balance of context coverage and response time
   - Captures multiple relevant passages
   - Still reasonably fast (4-4.5s average)
   - k=5 provides sufficient context for most procedural questions

3. k=10:
   - More comprehensive context but diminishing returns
   - 35% longer latency than k=1
   - May include some less relevant chunks
   - Can occasionally confuse model with excess information

4. k=20: (NOT RECOMMENDED)
   - Diminishing returns clearly visible
   - 75% longer latency
   - Often includes irrelevant context (scores below 0.5)
   - May dilute answer quality with noise
   - Waste of compute resources

5. Score Distribution Observations:
   - Top score typically 0.80-0.85 for good matches
   - Scores below 0.50 often less relevant
   - Gap between #1 and #2 indicates retrieval confidence
   - Large gap (>0.1): Very confident retrieval
   - Small gap (<0.02): Ambiguous query

6. Optimal k depends on:
   - Query complexity (simple queries need less context)
   - Chunk size (larger chunks need fewer retrieved)
   - Corpus characteristics (dense vs sparse relevant content)
   - Latency requirements

======================================================================
RECOMMENDATIONS
======================================================================

DEFAULT: k=5
- Best balance of quality and speed
- Sufficient for most RAG applications
- Reasonable latency (~4.5s)

USE k=3 WHEN:
- Very specific factual queries
- Low latency requirement
- Confidence in retrieval quality

USE k=10 WHEN:
- Complex synthesis questions
- Cross-document reasoning needed
- Latency less critical than completeness

AVOID k>10:
- Diminishing returns
- Increased noise
- Significantly longer latency
- Higher compute costs

CONSIDER DYNAMIC K:
- Use score threshold (e.g., include chunks with score > 0.6)
- Adaptive based on score distribution
- Stop when score gap exceeds threshold
